# Database Workflow and Source-of-Truth Rules

**Read this before making any change to the database or its access rules.**
This applies to the maintainer and to any AI assistant working on SQEPify.

---

## Why this document exists

SQEPify runs on Supabase. Supabase lets you change the live database by hand, through
the dashboard, in seconds: add a column, change a row-level security (RLS) rule, edit
a function. That convenience is also the trap. Every hand change that is not also
written down leaves the git repository describing a database that no longer exists.
Anyone reading the repository, human or AI, would then draw the wrong conclusion and
possibly act on it.

The rule that prevents this is simple to state, and the rest of this document is just
how to live by it.

> **The git repository is the single source of truth. The live database must never
> contain a change that is not also recorded here. If the two ever disagree, that is a
> fault to be fixed, not a normal state.**

---

## The files that hold the source of truth

| File or folder | What it is |
| --- | --- |
| `supabase/baseline/SCHEMA_BASELINE.sql` | A full snapshot of the schema at a known date. The reference of record. Regenerated by re-extraction, never hand-edited. |
| `supabase/baseline/EXTRACTION_QUERIES.sql` | The queries that regenerate the baseline. |
| `supabase/README.md` | A short map of the supabase folder. |
| `docs/SCHEMA.md` | Plain-English data dictionary: what each table is and who may read/write it. Does not list columns (the baseline owns those). |
| `docs/DATABASE_WORKFLOW.md` | This document. |
| `CHANGELOG.md` | The running record of every database change: plain English, the exact SQL that was run, and the SQL to undo it. Newest first, kept at repo root. |

If a change touches the database, it is recorded in the changelog with its SQL, every
time, without exception. If it changes what a table is for or who can see it, it also
updates `docs/SCHEMA.md`.

---

## How to make a database change (every time, no shortcuts)

1. **Get the exact SQL.** The change is written as a plain block of SQL, ready to
   paste, one logical change at a time. It is made safe to re-run: `IF NOT EXISTS`
   when creating, and `DROP POLICY IF EXISTS ...` immediately before `CREATE POLICY`.
   Pasting it twice should do no harm, because changes here are applied by hand.

2. **Apply it.** Paste the SQL into the Supabase SQL editor and run it. This project
   does not use the Supabase command-line tool; applying by paste is expected.

3. **Record it, with the SQL.** Add a dated entry to the top of `CHANGELOG.md`: the
   plain-English what and why, the exact SQL that was run, and the SQL to undo it.
   The changelog is the record. There is no separate migrations folder. If access or
   meaning changed, update `docs/SCHEMA.md` in the same breath.

4. **Keep the snapshot honest by re-extraction, not by hand.** Do not hand-edit
   `SCHEMA_BASELINE.sql` for each change. Periodically, after a batch of changes or
   before a review, re-run the extraction queries and regenerate the baseline from the
   live database so it is trued up from the source of truth.

5. **Commit the documentation together.** The changelog entry and any `SCHEMA.md`
   update go into git in one commit. Frontend changes deploy to Vercel on push.

---

## The single biggest source of drift, and the rule about it

Most drift does not come from running SQL. It comes from the dashboard's point-and-
click editors: changing a column in the Table Editor, toggling an RLS policy in the
Authentication screen, editing a function in the UI. These make a real change to the
live database and leave no trace in git.

> **Do not change schema, RLS, functions, triggers or storage rules through the
> dashboard's point-and-click screens. If you find yourself about to, stop, write the
> equivalent SQL, run that, and record it in the changelog instead.**

The only acceptable dashboard action for a schema change is pasting and running SQL
that you then record in the changelog with its undo.

## Instructions specifically for an AI assistant

For how the maintainer likes to work in general (tone, principles, how changes are
deployed), read `AGENTS.md` at the repo root first. The points below are specific to
database tasks.

- **Do not trust the dated baseline as a live picture if changes have been logged
  since.** Read `docs/SCHEMA.md` and the most recent `CHANGELOG.md` entries first; the
  changelog holds the SQL applied after the baseline. If anything is unclear or looks
  out of date, ask the maintainer to run the extraction queries and work from those
  results.
- **Produce changes as a plain block of SQL ready to paste,** with the safe-to-re-run
  pattern, plus a matching block of undo SQL. There is no migrations folder; the SQL
  is recorded in the changelog entry, not in a separate file.
- **Always supply the changelog entry (containing the SQL and its undo) and, where
  relevant, the `SCHEMA.md` update, in the same response as the SQL.** A change is not
  finished until it is recorded.
- **Never put secrets in any committed file.** No service-role keys, no provider API
  keys, no Control Room feed token. Those live only in the edge-function environment.
- **When you describe access rules to the maintainer, describe them in plain English**
  (who can read, who can write), because the maintainer does not read SQL.

## Re-baselining: rebuilding the snapshot from the live database

Do this when first adopting this workflow, and again if you ever suspect the repo and
the database have drifted apart. Run the queries in
`supabase/baseline/EXTRACTION_QUERIES.sql` in the SQL editor and save each result
(use the editor's Download CSV button so the full text is captured rather than copying
the cell, which can be truncated). Hand the results to whoever is rebuilding the
baseline, who reassembles them into `SCHEMA_BASELINE.sql`, refreshes `docs/SCHEMA.md`,
and commits both with a changelog entry noting the re-baseline date.

---

## A drift check you can run any time

To answer "has anyone changed the database outside this process," re-run the
extraction queries and compare the results against the committed baseline and the
changelog. Any difference means an out-of-band change was made. Reconcile it by adding
a catch-up changelog entry that records what was actually done, so the history becomes
true again. Do not simply overwrite the baseline and move on, because that hides the
fact that the process was bypassed.
